# 项目计划：AI造物引擎

> 用数字的精通，去解决物理的无能。
> AI不造物，AI写代码——代码造物。
>
> 2026年6月19日 立项 · v2.0 修订版

---

## 一、项目愿景

**一句话：** 建立一个以"代码生成→仿真执行→结果分析→修改代码"为核心递归循环的系统，先在窄域验证循环的有效性，再通过分层迭代扩展到复杂物理系统。

**最终形态：** AI以需求描述为输入 → 在代码空间中做千百亿次递归迭代 → 收敛到一套经得起所有已知约束检验的制造图纸/施工图纸 → 人类（或工具）按图建造。

**和传统"AI辅助设计"的根本区别：**
- 传统：AI给建议，人做决策
- 本计划：AI写代码→执行→验证→修改（循环），人定义标准和验收最终结果

---

## 二、核心机制

```
┌─────────────────────────────────────────────────────────────┐
│  递归迭代循环（AI造物的原子操作）                            │
│                                                              │
│  ┌────────────┐   ┌──────────┐   ┌──────────┐   ┌────────┐ │
│  │ AI写代码   │→  │ 执行代码  │→  │ 代码执行 │→  │ AI分析 │ │
│  │ (生成设计)  │   │ (仿真运行) │   │ 返回结果 │   │ 并修改 │ │
│  └────────────┘   └──────────┘   └──────────┘   └───┬────┘ │
│       ↑                                              │      │
│       └──────────────────────────────────────────────┘      │
│                      千百亿次迭代                             │
│                                                              │
│  最终：收敛到满足所有约束的代码 → 输出制造图纸/施工文件        │
└─────────────────────────────────────────────────────────────┘
```

**关键特性：**
- AI只做一件事：**写代码**。不画图、不拉滑块、不点按钮
- 反馈来源：**代码执行结果**（编译器/FEA求解器/CAM后处理器），不依赖AI自我判断
- 收敛保证：**确定性验证代码**作为迭代终点，不满足约束绝不停止
- 可扩展性：**同一循环在不同尺度运行**——从支架到飞行器，只是参数数量、仿真精度、分层数不同

---

## 三、分层迭代架构

千百亿次迭代不是在一个循环里完成的。按精度和成本分层：

```
层级  | 迭代次数  | 单次成本  | 作用
──────┼──────────┼──────────┼────────────────────
L1    | 10⁷-10⁸  | 毫秒级   | 用代理模型/解析解做粗搜索
      |           |           | 快速淘汰不可行域
──────┼──────────┼──────────┼────────────────────
L2    | 10⁴-10⁵  | 秒级     | 简化FEA/粗网格仿真
      |           |           | 缩小候选设计范围
──────┼──────────┼──────────┼────────────────────
L3    | 10²-10³  | 分钟级   | 高保真FEA/CFD
      |           |           | 精确验证最终候选
──────┼──────────┼──────────┼────────────────────
L4    | 1-3      | 天级     | 真实物理测试
      |           |           | 校准仿真偏差，最终签字
```

每层的输出是下一层的输入约束。**这和深度学习训练策略完全一致——先用廉价近似做大量探索（训练），再用高精度验证做最终确认（测试）。**

## 四、执行路线图

### Phase 1：验证循环（0-4个月）

**目标：** 选一个窄域，跑通"AI写代码→执行→分析→修改"的完整循环，打样验证。

#### 选域：注塑件支架（不变）

域的特征：参数少（~10）、仿真快（秒级）、打样便宜、经济价值明确。

唯一变化——**工具链围绕"代码生成"重新组织**：

```
新架构（代码中心）                旧架构（AI设计中心）
───────────────                   ───────────────
AI写CadQuery脚本                   AI"设计"支架
→ 执行生成STEP                     → 人转成CAD图
→ AI写FEA脚本（CalculiX输入）       → 人设FEA边界条件
→ 执行FEA                          → 人跑FEA
→ AI写分析脚本（读FEA结果）          → 人读FEA结果
→ AI写修改脚本（调参再生成）         → 人手动调参
→ 循环                              → 循环
```

核心区别：**每一环AI写的都是可执行代码。人的角色从"操作员"变成"验收员"。**

#### 里程碑

| # | 时间 | 完成标准 |
|:---|:---|:---|
| M1 | 第2周 | 环境就绪，第一个参数化支架的 CadQuery 脚本由AI写就并生成STEP |
| M2 | 第4周 | AI写的FEA脚本跑通，获得应力/形变结果 |
| M3 | 第8周 | 完整循环跑通：AI自主完成一次"生成→FEA→分析→修改→再生成" |
| M4 | 第12周 | 累积500+次循环，选最优设计打样并通过物理测试 |
| **M元** | **第16周** | **元检查：验证AI能否在"两个专业"场景下保持代码一致性（新增）** |

**M元是什么：** Phase 1结束后增加一个小型压力测试——选一个涉及两个专业协调的简单结构（如"带排热孔的支架"——结构+热），验证AI能否生成两个模块的代码并通过全局协调器检查一致性。这个测试的结果决定是否进入Phase 2。

---

### Phase 2：加速迭代（4-8个月）

**目标：** 建立代理模型层（L1），让迭代数量级从千次提升到千万次。

#### 关键工作

1. **代理模型训练。** 用Phase 1积累的500+ FEA结果训练一个快速预测模型（XGBoost起步）。输入：几何参数。输出：预测应力/形变/通过率。这个模型可以在毫秒级替代FEA做粗筛。

2. **L1+L2双层循环搭建。** L1（代理模型）做千万次粗筛 → Top 100 进入 L2（简化FEA）→ Top 10 进入 L3（高保真FEA）。

3. **第一个"代理模型 vs FEA vs 真实"偏差数据集。** 每个最终打样的设计都记录三次结果：代理模型预测、FEA结果、物理测试结果。这个偏差数据集本身就很有价值。

#### 里程碑

| # | 时间 | 完成标准 |
|:---|:---|:---|
| M5 | 第20周 | 代理模型精度达到：通过/不通过预测率>85%，应力MAE<±15% |
| M6 | 第24周 | L1+L2+L3三层循环跑通，迭代速度提升100倍 |
| M7 | 第28周 | 偏差数据集积累到100+条，误差模式初步可视化 |
| M8 | 第32周 | 双层循环收敛的设计打样验证，一次通过率>70% |

---

### Phase 3：多模块扩展（8-14个月）

**目标：** 从单一零件扩展到多零件协调的简单装配体，验证全局协调器架构。

#### 关键工作

1. **定义模块接口协议。** 每个设计模块的输出必须包含形式化接口定义：几何边界、物理接口（力/热/流体）、安装约束。

2. **搭建全局协调器。** 确定性代码，读各模块的输出接口，执行碰撞检测+约束检查+一致性验证。

3. **验证场景：** 一个简单的三零件装配体——底座+支架+散热片，结构+热两个专业协调。每个零件由独立的AI代码模块生成，全局协调器验证它们之间是否冲突。

#### 里程碑

| # | 时间 | 完成标准 |
|:---|:---|:---|
| M9 | 第36周 | 模块接口协议定稿，第一版全局协调器代码跑通 |
| M10 | 第42周 | 三零件装配体的AI代码生成+协调验证+物理打样成功 |
| M11 | 第48周 | 发布v1.0开源版本：包含完整工具链、偏差数据集、文档 |

---

### Phase 4：系统级（14个月+，愿景）

**目标：** 扩展到更复杂的系统，最终向"完整制造图纸生成"迈进。

不再设具体里程碑——这部分依赖于 Phase 1-3 的实际进展。方向性目标：

| 系统复杂度 | 预计参数规模 | 需要先验证 |
|:---|:---|:---|
| 多零件装配体（10+零件，3专业） | ~10³ | Phase 3 全局协调器有效 |
| 建筑单层结构+围护+设备 | ~10⁴ | 多专业协调验证通过 |
| 飞行器部件（机翼段） | ~10⁴ | 分层迭代架构+耦合物理场仿真 |
| 完整建筑/飞行器 | ~10⁵ | 以上全部+计算资源到位 |

## 五、技术栈

### 代码生成层（AI在做的）

```
当前：Claude（写代码+编排整个循环）
未来：任何代码生成模型可替换此角色

AI生成的代码类型：
├── CadQuery Python脚本（三维几何）
├── CalculiX 输入文件（FEA仿真）
├── Python 解析脚本（读FEA结果）
├── Python 优化脚本（参数调整）
├── 模块接口定义 JSON（全局协调器输入）
└── 制造输出（G-code/工艺参数/施工图）
```

### 代码执行层（工具在做的）

| 功能 | 工具 | 说明 |
|:---|:---|:---|
| CAD生成 | CadQuery + OpenCASCADE | Python原生，参数化能力强 |
| 结构FEA | CalculiX | 开源，适合线性/非线性结构分析 |
| 热分析 | Elmer 或 CalculiX 热耦合 | 开源多物理场 |
| 流体 | OpenFOAM（Phase 3+） | 开源CFD行业标准 |
| 代理模型 | XGBoost → 小型神经网络 | Phase 2开始 |
| 全局协调器 | Python 自建 | 确定性代码，校验模块接口一致性 |

### 数据层

| 数据类型 | 存储 | 说明 |
|:---|:---|:---|
| 几何参数 + FEA结果 | SQLite / HDF5 | 结构化的迭代历史 |
| 代理模型 | ONNX / Pickle | 可导出可复用的预测模型 |
| 偏差数据集 | SQLite | 仿真vs物理的对比记录 |
| 模块接口定义 | JSON Schema | 各专业模块的接口规范 |

## 六、风险评估（v2.0 新增）

| 风险 | 概率 | 影响 | 应对措施 |
|:---|:---|:---|:---|
| AI在大尺度代码库中丢失一致性 | 🔴高 | 🔴高 | Phase 3前置验证（M9-M10），先从小型多模块系统验证 |
| 仿真误差在长因果链中累积不可控 | 🟡中 | 🔴高 | 误差显式建模+薄弱环节真实校准+保守退让策略 |
| 代理模型的预测偏差误导迭代方向 | 🟡中 | 🟡中 | 代理模型仅用于粗筛，最终判决权始终在FEA层 |
| 选定的域经济价值不足以支持持续投入 | 🟡中 | 🟡中 | Phase 1（4个月）就是价值验证器，不行换域 |
| 制造环节（打样）成为迭代瓶颈 | 🟡中 | 🟡中 | Phase 2开始用3D打印机本地快速迭代，最终验证才发工厂 |

## 七、立即行动（接下来7天）

```
第1天：安装 CadQuery + 验证能生成第一个简单支架STEP
第2天：安装 CalculiX + 验证能对一个简单模型跑通FEA
第3-4天：让AI写一个完整的"生成→FEA→读结果→修改→再生成"脚本原型
第5天：跑通第一个完整循环（不要求优化质量，只要求循环跑通）
第6天：记录第一个循环的数据格式，确定数据库schema
第7天：回顾一周成果，调整计划
```

**不要求完美——只要求循环跑通。** 一旦你亲眼看到"AI写代码→执行→分析→修改"的闭环在跑，你对这个方法的信任就不再是理论上的。
