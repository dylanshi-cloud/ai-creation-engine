# 架构 · 以代码为中心的物理AI系统

> 版本：v0.2 — 以"代码是精确交集"为核心重构
> 基于《两个不完美的头脑》第六章讨论

---

## 核心原则

> **AI不造物，AI写代码——代码造物。**

每一层的输入是代码，输出是可执行代码或代码执行结果。AI从不直接操作物理世界——它操作的是**控制物理世界工具的语言**。

## 各层详解

### 第0层：代码生成层（AI在做的）

AI（当前是Claude，未来可以是任何代码生成模型）接收"需求描述"，生成领域代码：

```
"一个L型支架，85×45mm，壁厚3mm，承重50N，材料ABS"
    ↓
Claude 生成：
  1. bracket.py (CadQuery脚本 → 生成STEP几何)
  2. bracket_fea.py (CalculiX脚本 → 运行FEA验证)
  3. bracket_analyze.py (后处理脚本 → 分析结果)
  4. 如果FEA不通过，修改参数重新生成1-3
```

AI永远是写代码的人。它不画图、不拉滑块、不点按钮——它写代码。

### 第1层：代码执行层（确定性工具在做的）

代码交给确定性执行引擎：

| 代码类型 | 执行引擎 | 输出 |
|:---|:---|:---|
| CadQuery脚本 | CadQuery + OpenCASCADE | STEP/STL 3D模型 |
| FEA输入脚本 | CalculiX / OpenFOAM | 应力/形变/流场数据 |
| 分析脚本 | Python / NumPy | 通过/不通过、优化参数 |
| 制造脚本 | CuraEngine / CAM | G-code / 刀路文件 |

这一层没有任何AI。它是"物理世界的编译器"——输入代码，输出精确物理结果。

### 第2层：迭代层（AI+代码在做的）

这是核心创新点。AI不靠"推理"来优化设计——它靠**写代码来分析代码执行结果，然后修改代码**：

```
① AI写生成脚本 → ② 执行 → ③ AI写分析脚本 → ④ 执行分析
→ ⑤ AI根据分析结果写修改后的生成脚本 → 回到②
```

这个循环的本质是：**AI在代码空间中做梯度下降。** 每一步的"梯度"来自代码执行结果，不来自AI的猜测。

### 第3层：制造输出层（最终交付）

收敛后的设计输出为制造指令：

- 3D打印 → G-code（CuraEngine自动切片）
- CNC加工 → 刀路文件
- 注塑 → 工艺参数卡 + 模具图纸规格
- 装配 → 装配序列代码

## 和传统AI架构的关键区别

| 传统思路 | 本架构思路 |
|:---|:---|
| AI需要物理世界模型 | AI需要代码生成能力 |
| 需要百亿级仿真数据预训练 | 需要代码模板和API文档 |
| AI"理解"物理规律 | AI生成调用物理引擎的代码 |
| 核心瓶颈：物理数据 | 核心瓶颈：代码正确性 |
| 验证方式：物理测试 | 验证方式：代码编译+运行 |
| 人类角色：标注数据 | 人类角色：验证代码输出 |

**后者每一步都是AI已经证明过自己能力的领域。**

---

## 为什么这个架构能启动

因为不需要等待任何技术突破。

- AI写代码的能力已经就绪（Claude/GPT-4级别）
- CadQuery/C计算库的API已经就绪
- FEA引擎已经就绪（几十年的工业验证）
- 代码执行环境已经就绪（Python/Shell）

**唯一需要搭建的，是把这些串起来的管线——而这条管线本身，也是在写代码。**

---

## 迭代策略

```
Phase 1 (0-3个月):
  单域闭环：AI写CadQuery脚本 → 执行生成模型 → AI写FEA脚本 → 执行验证 → AI修改 → 收敛
  → 验证代码闭环可以替代"物理直觉"

Phase 2 (3-6个月):
  数据积累：每一轮(代码→执行结果→修改)都是训练数据
  → AI学会"哪些参数组合大概率通过FEA"
  → 减少迭代轮数

Phase 3 (6-12个月):
  域扩展 + 制造输出
  → 新域只需要新的代码模板
  → 最终输出G-code/制造指令
```

---

## 这个架构的验证标准

简单直接：**代码能编译通过并执行到预期的结果。**

这不是一个需要"实验验证"的假设——每一天的代码运行都在验证它。
